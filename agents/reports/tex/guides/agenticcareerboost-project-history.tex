\chapter{Purpose}

\takeaway{This report is the bridge between the public project page and the
deeper evidence body. It tells the development history without hiding the
failure modes that made the current architecture necessary.}

AgenticCareerBoost is not only a static portfolio page. It is a repository-backed
record of agentic development: rules, state, logs, reports, site routes, PDFs,
validation scripts, and human-owned external tasks.

The website should make the project legible quickly. The LaTeX report set should
carry the heavier proof: what was built, what failed, how the system recovered,
and which evidence supports each claim. This document maps those two surfaces
together so future social posts can reuse the same narrative without inventing a
new story. The first LinkedIn presentation has now happened as a human-owned
external action; the next public work is to unpack the deployments, structure,
history, and fixes behind the system.

\chapter{Current Public Story}

\takeaway{The site should explain the project as a traceable engineering
artifact. Social posts should distribute that evidence, not replace it.}

The current story is simple enough to reuse across the website and later social
posts:

\begin{quote}
AgenticCareerBoost began as a path-based agentic workflow and grew into a public
proof repository. It then hit the same failure mode many agentic systems hit:
too much stale context, too many active meanings, and not enough authority
separation. The current version is valuable because it records the cleanup as
part of the work: rules define behavior, state records evidence, reports explain
the arc, and the site exposes the proof.
\end{quote}

That is why the project page should expose all formal reports and not only the
current dashboard. The dashboard tells the present state. The report library
shows the development history.

\chapter{Post-Launch Hardening}

\takeaway{The first presentation made the next problems clearer. The fixes are
part of the project proof because they show where agentic workflows fail when
deployment, authorship, and validation boundaries are weak.}

The first presentation was not followed by a clean victory lap. It exposed sharper
engineering problems:

\begin{itemize}
  \item nested GitHub Pages routes escaped from \safePath{/AgenticCareerBoost/}
  to root paths;
  \item direct PDF links could bounce between \safePath{/files/...pdf} and
  \safePath{#/files/...pdf} when the deploy artifact was missing;
  \item the CV and cover-letter pipeline depended on generated files and fixed
  names instead of a manifest;
  \item generated tests overreached into authorial LaTeX content decisions;
  \item validation overfit hotfix assumptions instead of testing durable
  contracts.
\end{itemize}

The correction was to narrow the contracts: app routes are app routes, static
files are static files, CV artifacts are manifest-declared build outputs, and
tests verify integration rather than authorship. That is the story S-005 should
carry forward.
