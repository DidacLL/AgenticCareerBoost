NOTE

Why this site is intentionally simple

This site is static HTML, CSS, a little JavaScript, and a few JSON files. That is not nostalgia. It is a constraint.

A personal site used as a public technical surface should be easy to inspect, cheap to host, hard to break, and honest about what it is doing. I do not need a stack that needs its own incident response plan just to say “here are my projects”.

The funny part is that simple files make agent-assisted work safer. An agent can edit a static page, but the change is visible as a diff. The validator can check routes. The browser can show the result. The repo keeps the memory. Nobody has to believe the tool because it sounded confident.

TRADE-OFF

What I gain and what I lose

I lose some convenience: no CMS, no automatic content pipeline, no fancy component system. I gain something I value more right now: the ability to open the files and understand the whole surface without needing a private tour through the build stack.

That makes the site a workbench. It can hold CV views, project notes, blog entries, generated indexes, and manual corrections without pretending to be a product. It is allowed to be personal because it is mine. The professional part is that it still needs to compile, route, and make sense.
