What this section is for

This is not a content machine. I am not trying to cosplay a media company from a GitHub Pages site. The blog is a shelf for writing that has a source behind it: LinkedIn publications, sprint reviews, project notes, and short technical decisions that are easier to read here than inside a repo tree.

When a LinkedIn post is published, this page can keep the calmer version: more context, less algorithmic theatre, and a link back to the public artifact. When a sprint closes, this is where I can explain what changed without forcing a recruiter to read a closure matrix like bedtime literature.

RULE

How posts get here

A post needs one of three anchors: a published external link, a project page, or a sprint/repo artifact. Otherwise it stays a note. Empty essays are easy to generate and hard to respect.

The source list lives in blog-index.json. The boring file is part of the point: if the site says something exists, there should be a path to it.
