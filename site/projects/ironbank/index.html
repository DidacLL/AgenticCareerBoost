01C / PROJECT

What it is

IronBank is a Java/Spring Boot banking simulation from my Ironhack period. It is older work, kept public because it shows the backend route before the current agentic/ML systems direction.

It also connects to my operations background. Accounts, transactions, permissions, rules, edge cases, and process failures are not abstract vocabulary when you have spent years around banking and insurance systems.

WHY

Why I built it

The project connected banking-domain familiarity with software structure: users, accounts, transactions, rules, persistence, and service boundaries.

It was an early backend artifact and a useful baseline for seeing how domain rules move from operational language into code.

HOW

How it works

The application uses a REST-oriented Spring Boot structure with persistence and banking-domain entities.

The current value is not polish. It is a visible baseline: where the backend path started, what I would document better now, and why domain behavior should not disappear into controller logic.

WHERE IT FITS

Why it is still here

IronBank is not the flagship. AgenticCareerBoost is. Keeping the older work makes the route legible: regulated operations, backend systems, tooling/documentation, agentic workflows.

The continuity matters more than pretending every public artifact is current-center work.

LEARNED

What it taught me

Domain knowledge helps, but code still has to make rules explicit, testable, and maintainable.

Backend work is most interesting when it carries behavior: constraints, transitions, failures, permissions, and state changes.
