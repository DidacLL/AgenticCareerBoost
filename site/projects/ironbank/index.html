01C / PROJECT

What it is

IronBank is a Java/Spring Boot banking simulation from my Ironhack period. It is not my newest work, and I do not want to pretend it is. I keep it public because it shows the backend route that started before my current agentic/ML systems direction.

It also connects to the part of my background that still matters: banking and insurance operations. Accounts, transactions, user flows, permissions, rules, edge cases, and boring domain constraints are not abstract to me. I spent years around systems where a small process mistake can become a very real human problem.

WHY

Why I built it

At the time I wanted to connect what I already knew from banking operations with software structure: accounts, users, transactions, rules, and the kind of domain details that become annoying if the model is too casual.

It was also a useful early backend artifact: Java, Spring Boot, REST structure, persistence, service boundaries, and the usual lesson that naming things is somehow still harder than everyone pretends.

HOW

How it works

The project is a Spring Boot application with REST-oriented structure, persistence, and banking-domain entities. The value today is less in polish and more in showing my older backend base plainly.

I would not present it as a finished product. I would present it as a visible baseline: this is where the backend path came from, this is what I would now document better, and this is the kind of domain model I prefer to make explicit instead of leaving buried in controller logic.

WHERE IT FITS

Why it is still here

IronBank is not the flagship. AgenticCareerBoost is. But removing older work just because it is older would make the profile less honest. I would rather show the route: regulated operations → backend systems → tooling/documentation → agentic workflows.

That path is more useful than pretending I appeared from nowhere holding a perfect portfolio and a ring light.

LEARNED

What I learned

Domain knowledge helps, but software still demands its own discipline. Knowing banking terms is not enough; the code has to make the rules explicit and testable.

I also learned that backend work becomes much more interesting when it is not treated as CRUD decoration around tables. The useful part is the behavior: constraints, transitions, failures, and who is allowed to do what.
